這幾天試了一些免費燒錄軟體，包括 CDBurnerXP、ImgBurn、InfraRecorder 等。種種因素下，我還是灌了 Nero 來用。習慣果然是非常根深蒂固的。不過這個軟體安裝了一些舊的系統更新，而我可能因為昨天利用 Windows Update Remover 刪除了所有系統的更新備份，造成系統即使顯示安裝成功，總是會要求重複安裝某些更新。結果工具列上的更新圖示就無法消除了，看來有些礙眼。最後依 Microsoft Answers 建議用上 .NET Framework Cleanup Tool 才解決。前幾天買的那一臺光碟機，最低也有六倍數，燒一片約需要十一分鐘。雖然使用的時間比較少，但常常會讓燒錄程式掛掉。這個問題看起來像是系統層面的問題，因為不論是 ImgBurn 或者 Nero 都會有祭出了工作管理員卻砍不了程式的問題。但是只要一將 USB 外接式燒錄器的電源拔掉，程式就能回復正常。因為狀況頗為頭痛，我甚至開始考慮是不是要退貨了。
語音輸入的部分，正如前幾天所言，準備開始大量投產。雖然以說話的方式輸入而不是唸稿對我來說還不太習慣，速度上大約只有打字的 ⅔ 到 ½ 左右（應該更快一點，這是低估了。），但最起碼今天的日記已經可用語音的方式輸入。這幾天學到的技巧是，在做更正的時候，以滑鼠輔助選擇，而不直接用念的。這樣似乎能夠增進輸入的效率。因為在更正框中，有時念數字卻會被當作是語音輸入。輸入時常常會在這上面浪費許多時間。（後來才發現，數字之後還得要加一個「號」才行。果然是我不會用？）以滑鼠輔助的話，能改進輸入的速度。由於對某些特定的詞彙，偶爾唸了好幾次還是出不來，只好以意思相近的詞彙作替換；若是候選清單能夠列出相同意思的詞彙，應該也不錯。另外，firefox 的輸入框似乎常常語音輸入衝突，一點選就會讓 firefox 掛掉。暫時以 Chrome 搜尋來解決。

「略」同「掠」兩字之使用原則

意外發現中文只有「攻掠」，「攻略」在教育部辭典中查不到，看似日語的用法。但我過去文章竟沒出現過任何的「攻掠」。雖然汉典提到後漢書 志第十一 天文中也有用過這個詞，並且經查似乎如此，但終究無法斷定正確用法。再查一下自己過去的用法，卻發現一樣有問題的是「侵略」與「侵掠」：我竟從沒用過侵掠這詞。花了兩三小時瀏覽了一下，雖然不能說明白了，但總是有些心得，小記一下結論。
「略」、「掠」兩字在康熙字典中提到時，可以發現解戰國策「掠於郊野」時已「通作略 亦作剠 或作擽」，而略則是在解左傳宣十五年時出現「取」這個比較接近「掠」的意思。掠在說文解字作奪取，算是比較接近現時理解的。但略在說文中僅為經略土地，或解作經營吧；那麼略到底何時開始通掠，甚至較掠常用於「奪取」之意的呢？查了好些資料，終於在陰山高闕與陽山高闕辨析發現些許苗頭。原來略字本義當為田地的界限。如此，我就猜想左傳中的用法也不過是種引申了。簡單的說，略由原意引申，是用在侵犯、占領他人領土等土地的時候；掠則便是「奪取」之意，無論是財物或是人畜。
個人是很想照這個原則使用這兩個字，但這樣一來就會發生許多困擾。日文中對「遊戲攻略」的使用還只是小事，即使在中文，這兩字用於「侵犯掠奪」意也已通用許久。將略用在奪犯土地之外的不計其數。見教育部辭典可發現，侵略與侵掠不只是同時皆有「侵犯掠奪」，甚至侵略都擴大釋義到對主權及政治的侵犯了。在這種情況下，我想我也只能在較無公認用法的時機採取此原則罷了。而實際上，結果恐怕還是多以「略」為主了吧。

從 2010/1/22 就開始利用 Git，但直到昨天洗澡時才想到以 Git 來作更新。之前都用 FTP，流程如下：
  1. 在 notebook 編輯文稿
  2. 與 PC 作 synchronize
  3. sftp 到 server 上
まあ、其實要問我到底需不需要版本管理系統，這問題也有點妙。其實不管是文件或是 code，我長期以來都是使用檔案內履歷的方法：東西只會多不會少（尤其是對文章這種東西），真的要改（或大段的改），舊的東西會以註解的形式放在原先的位置。（我常用 insert 或 replace，東西很多時用 append，幾乎不用 delete。）以這些文字來說，真的會作替換的主要是我自己寫的詞彙校對程式。而在這部分，我主要是以每隔一段時間（半年至一年吧，以十年來的頻率來說。）作完全備份一次。因此整體說來，只要留著最新的那一份也就夠了。
為了達到更方便的目的，總須多看一點文件。另參考了 How to Use Git，可惜結果還是不盡如人意。我的理想是以 Ctrl+s 儲存檔案後就能自動更新，差一點的話每天午夜進行自動更新也還能接受，而今天找到的看來實在太過麻煩了。
我的目標和普通的不同。我希望以 PC 或 notebook，而不是 server 上面的 repository 作為主幹。一開始當然是在 remote server 作 `git init`。然後於 notebook commit 會碰到 "updating the current branch in a non-bare repository is denied"。這勉強以 `git config receive.denyCurrentBranch ignore` 解決了（我不想用 `git --bare init`，這會將一籮筐設定檔與設定目錄直接倒在 repository 中，不容易分辨。），但後面的 non-fast-forward updates were rejected 則相當棘手，非得加上個 --force 不能處理。而即便這些都不算，還得加個 post-script 才能更新；就算要加在 crontab 中都嫌累贅。也就是說，當下以 Git 來說，要與伺服器同步的流程是：
  1. 在 notebook 編輯文稿
  2. 用 TortoiseGit 等工具與 server 上的 repository 同步（其實光這個動作就包含許多步驟，差只差在可以更快的作反應而已。），需要勾選 --force，否則就得再來一次。要記得這一步，對懶人我來說當然是很麻煩的。
  3. ssh 到 server 上 `git checkout -f && chmod o+r ./files`
  4. 隔段時間再和 PC 同步。
這樣下來太不夠自動化了，只能再看看有沒其他好方法吧。TortoiseGit 應該加上 comment 完自動 synchronize 或 push 的功能吧。當然你可以罵我自找苦吃，不用 blog 或 wiki 之類的現成省事辦法。但對於常常需要把舊文章拿起來校對考訂，並且早已為此發展了一點小工具的我來說，終究還是擺在本機上會比較方便啊。
server 零時起當掉了。說不定是每天例行性工作造成的問題。會直接掛掉，大概和利用到舊系統那塊硬碟脫不了關係？

我們難逃的劫數

好吧，今天要說的這個，就算不是陰謀論，也可謂是危言聳聽了。
前些日子提過郭凯的評論與 ±2°C 和極端天氣 global weirding 議題。洗澡時又回想到最近美國的量化寬鬆，以及我雖然在一兩年前就懷疑歐洲（某些）人過的優渥生活與他們其實工作太少、享受太多有關（雖然這有可能只是空想），但我該不會錯過了認知的時機：或許美國也是如此的，因此才需要如此大幅度的調整？既然是這樣，我是不是應該想得更遠一點？
接著我覺得我們所欠缺的，並不是要有、以及有多少人跳出來說「你們太奢侈了！」；而是能使他們這些人（或是說我們）理解，自以為普通的生活水平，在永續的角度下實際上是奢侈，並能立而行這件事。而實際的解決策略，可能是減少人口、控制生育等。無論我們多麼節儉，十個克制的人還是吃得比一個稍微奢侈的人（像是現在的我們）多。我回想到前幾天才被問到我為何贊同墮胎。雖然當時只是恍惚中隨便回應一下而已，但這畢竟是公說公有理，婆說婆有理的問題；我回答若不能提供合適的教養環境，何生之？擴大來說，這就是一種避孕了。
以這角度來說，印鈔票真是苦肉計。或許 Bernanke 知道他們的問題不在讓大家又能高高興興地花錢買東西，而是該怎麼跟他們說「你們別買那麼多高科技產品、新衣以及垃圾食物了！你們不能買了。」。或許他們不知道。無論如何，我覺得讓大家懵懵懂懂中再高興個或許是幾年，然後再來面對不願意正視的真相（不只是指 Al Gore 那一部啦。全球暖化其實只是其冰山一角。重點是我們的態度。），其實是很艱難的決定。由奢入儉難（棘輪效應），而我們又把「維持在奢」當作優秀執政的證明，不願意得罪民心。不，應該說膽敢得罪，總會換上另一批不去得罪的。我們都知道，不能光是以慾望改變現實，真相也無法由投票決定。民主不代表我們可以投票表決每個人都多個一噸黃金擺家中，或要求政府多給每個人一萬億美元。但我們不願意竟低自己的生活水準，其實不過是種變相的耍性子罷了。只不過沒多少人願意察覺。
以這角度來說，我們甚至等不到第三世界跟上我們的步伐；那快要成為不切實際的夢想了。非洲要能追上我們？現實是我們會被拖下去。

或許你還是會問我（雖然聰明如你早就知道了），為何不樂觀一點呢，想些正面的可能性呢？很遺憾的，以我之前對人類的整體印象來說，即使有很多人在亟聲呼籲了，我認為人類終將引發爭端。無論是為了純淨淡水、糧食等生活必需品的戰爭，或是其他能源資源的強奪；就算我們不會明爭，也會暗鬥。當大陸貨印度這樣的國家也急起直追西方的消費形式時，我們又能何如？我們又不能責難他們。我得老實說，因為連我自己都不願意降級回去過苦日子啊！身上只有破布衫、要吃只有番薯籤，我們都看不起遊民乞丐了，難不成自己還要變成他們麼？見苦しい。所以你還期望人們會怎麼「自主」的抉擇呢？
以這角度來說，我們（的下一代）真的是生於安樂、死於憂患。因此說要預防或阻止，我倒寧可預測在那之後的人間景象。

但人世又是如此的幻化無常，世上到處都是巴西的蝴蝶，要準確預估未來世界的模樣猶如要報導一年以後的天氣。

  1. 第一層次是就作品的內容評論與辨正。以小說如紅樓夢為例，無非寫寫劇情簡介、照劇情領著跑一次，談談腳色心理與作者如此處置的用意，然後說，啊、林黛玉如何令人憐愛、假如寫成個大團圓多好，以至這邊出紕漏、那邊有缺點、曹雪芹對寶玉情有獨鍾等。大意、分析腳色行為動機與劇情設定、結構、詞藻等屬此類。（不用說，操縱性之類的尚不入流。）
  2. 第二層次是探出作者真義。上面是以讀者的身分來讀作品的話，這邊開始是用作者的角度看待。包括訓詁、寫作背景、作者為何會寫出這樣子的作品都涵蓋在這個層次內。
  3. 最後是跳脫作者與作品。簡單的說，上一個層次是與作者平起平坐的話，那麼這邊我們就要再進一步，超越作者，引入一個更圓融的體系、或料見一個作者未想到，作品中也沒能展現給我們的世界。當然這話是說大了。不過三人行必有我師焉、諸葛亮未必完勝臭皮匠，作者與作品不見得能覆盡一切，這時就輪到我們來補充了。前面尚屬融會貫通（內化至自己原有思想體系），之後還需學以致用。完善作者於作品中的體系、擴大到現實以及古今中外（真實）環境並作深入探討歸於此類。
當然這不能說是分類，也無法完善的組織任何評論。更不能說是終極、或是最佳。但就如同文體沒有絕對的分類法，我想就我來說這樣子的層次暫時已經夠了。以此角度來說，好的評論能予人甚至比作品本身更深遠的意涵。字數倒不是最大問題，雖然要把自己的意思闡述清楚通常已有相當分量。這或許已經太過高估評論，甚或給人過分的重荷；但的確有些評論能達到這樣的高度，使得一個差勁的作品卻依舊能（於看完評論後）發人深省。
至於評分系統，經過 YouTube 的實證，現代人似乎大多只喜歡好惡二分法，分等級是沒啥用的。或許是因為大家都很忙、沒時間，或者人們漸漸沒有這習慣了？（不用說我已經想改造本站的評分制度想很久了，無奈心有餘而時間不足。←だから、こんなのは言い訳と言ってるの！）
分類標籤的話，這邊也思考很久了。不過一堆標籤擺在一起還是很礙眼，要實作的話應該會用樹狀結構來分類才是。大概就像是先大類分作所探討（哲學）議題、推薦對象、劇情型別等等幾大類，各能覆蓋全部主題。其下再作細分之類的。

當 reference 作了變動，修改完顯然這流程亦需要做更動以適應其差異。不用說，若是原始資料（例如法條）就是以 SOP 之類 production environment 可直接適用的形式存在會更好；但實際上這些資料常常是以文件的模式存在，因此最起碼得建個 API 之類的 interface 做轉換。依照現行常見編輯模式，根本只能手動依照（可能還是第三方，如是政府出版的法條）差異文件增修；但這得一條條對照，實在很麻煩。並且，事實上很多其他領域說明文件也有這樣的情況：糟糕的一點都不知道怎麼來的。好點的會加 link 可以溯源，但當所引用的原始依據發生變化時，這些 referent 不一定都能隨即更新。我突然想到，這應該可以更為自動化：我們可以設計一套演算法，取得原始依據的特徵值（如 hash 或文字 concat——這可能在有必要維護副本的情況用到。）。然後再訂定一個機制，使 referent 可以察覺變化，甚至自動 adapt。這機制包括主動例如 feed，或是被動例如利用排程執行偵測程式；以及警示程序，例如出現變更就警告編輯者有需要做更動，最起碼能告知使用者原始來源已經有所變更。要自動 adapt 可能還需要比光是取得特徵值更強的演算法，可以以之得到標的應如何變更的模式（已預先定義的函式、法則等）。而不能取得模式的情況，當然還是得人工手動變更。這樣應該就能更為方便的做到流程的更新。
